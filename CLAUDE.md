# CLAUDE.md - AI 开发约束文档

> 本文档用于约束 AI（Claude/Trae 等）在处理本项目时的行为规范和输出标准。
> 任何 AI 助手在参与本项目开发前，必须先阅读并遵守本文档。

---

## 🎯 核心原则

### 1. 需求驱动
- **任何代码修改前，必须先阅读对应模块的 MD 需求文档**
- 不理解需求就写代码是严格禁止的
- 对需求有疑问时，先向用户确认，不要假设

### 2. 结构一致性
- 新增模块必须遵循现有文档结构和代码结构
- 不要引入新的目录结构或命名规范，除非用户明确要求
- 保持代码风格与现有代码一致

### 3. 简洁优先
- 单个函数复杂度不超过 20 行（不含注释和空行）
- 做好函数解耦，一个函数只做一件事
- 避免过度工程化，先实现核心功能

### 🔍 步骤1：获取文档

**必须读取的文件**：
- 对应模块的需求文档（如 `ERP_需求文档/01_销售管理.md`）
- 前端预览代码（如 `ERP_前端预览/index.html`）

**读取目的**：
- 了解当前模块已有的功能和结构
- 确保新需求与现有内容兼容
- 保持文档风格和格式一致

### 📊 步骤2：分析对比

**对比维度**：

| 对比项 | 分析内容 | 处理方式 |
|--------|---------|---------|
| 功能重叠 | 新需求是否与现有功能重复 | 合并或覆盖，需用户确认 |
| 字段冲突 | 同一字段的定义是否矛盾 | 列出差异，请用户选择 |
| 流程变更 | 是否修改现有业务流程 | 明确变更范围和影响 |
| 新增功能 | 完全独立的新功能 | 直接添加 |
| 遗漏补充 | 现有功能的不完整描述 | 补充完善 |

### 💬 步骤3：需求对称

**必须确认的情况**：

| 场景 | 必须询问的内容 | 询问方式 |
|------|--------------|---------|
| 需求描述不清晰 | 补充业务场景、使用者、触发条件 | 提供具体示例 |
| 需求存在歧义 | 明确用户意图 | 列出所有可能的理解供选择 |
| 与现有文档矛盾 | 确认以哪个为准 | 指出矛盾，说明各自优缺点 |
| 涉及分期实现 | 每期的功能边界 | 用表格列出分期计划 |
| 涉及技术实现 | 实现方案选择 | 说明各方案特点，不替用户决定 |
| 涉及其他模块 | 跨模块影响范围 | 列出涉及的模块和数据流转 |

**对称原则**：
- 一次最多 4 个问题，避免信息过载
- 问题要具体、有选项、有上下文
- 使用表格和列表让问题清晰
- **未确认清楚绝不进入下一步**

### 📝 步骤4：合并更新文档

**合并原则**：
- 新增功能：添加到对应章节末尾
- 修改功能：覆盖原内容，注释说明修改点
- 删除功能：用删除线标注，等待用户确认

**文档更新格式**：
```markdown
## 本次更新说明（新增）

### 2026-05-25 更新
- 新增：XXXX功能（用户输入来源）
- 修改：YYYY字段定义（原定义 → 新定义）
- 补充：ZZZZ业务流程说明

---

## 正文内容（保持原有结构）
...
```

### 🎨 步骤5：按需渲染前端预览

**渲染条件**（满足任一条件即可）：
- 用户明确要求查看预览效果
- 新增或修改了页面结构
- 需要可视化确认功能流程
- 用户要求验证需求理解是否正确

**不渲染的情况**：
- 纯文档整理，不涉及页面变更
- 用户未要求预览
- 仅补充说明性内容

#### 3. 前端预览规范
前端预览页面必须遵循以下原则：

| 原则 | 说明 |
|------|------|
| 简单交互 | 仅实现最基本的页面切换，不做真实的数据交互 |
| 粗粒度原型 | 只展示页面结构和关键元素，不追求像素级细节 |
| 纯静态展示 | 不依赖后端 API，所有数据用硬编码模拟 |
| 快速验证 | 目的是让用户快速确认页面布局和流程是否符合预期 |

**技术约束**（已存在）：
- 使用纯 HTML + CSS + JavaScript
- 不使用 Vue/React 等框架
- 不使用构建工具，浏览器直接打开即可运行
- 参考 Ant Design 设计规范

**动态效果标注规范**：

当某些需求效果难以通过静态页面直接展示时（如弹窗、联动、异步加载等），必须使用**标注框**的形式在页面上说明预期效果：

| 需要标注的场景 | 标注内容示例 |
|---------------|-------------|
| 弹窗效果 | 【弹窗说明】点击后打开新建订单弹窗，自动填充当前订单的客户信息 |
| 联动效果 | 【联动说明】选择客户后自动带出联系人、电话字段 |
| 异步加载 | 【异步说明】商品列表支持搜索，延迟加载商品数据 |
| 条件显示 | 【条件说明】仅在订单状态为"已审核"时显示"生成出库单"按钮 |

**标注框样式要求**：
```css
.annotation-box {
    background: #fffbe6;      /* 浅黄色背景 */
    border: 1px solid #ffe58f; /* 黄色边框 */
    border-radius: 4px;
    padding: 12px 16px;
    margin: 8px 0;
    font-size: 13px;
    color: #d48806;            /* 深黄色文字 */
}
```

**标注框位置**：
- 紧跟在触发元素下方或旁边
- 保持与周围元素的视觉关联性
- 不遮挡主要内容

### 需求文档编写规范

当为用户编写或修改需求文档时：

1. **使用表格描述列表字段**
   ```markdown
   | 字段名 | 类型 | 说明 | 宽度建议 |
   |-------|------|------|---------|
   | 订单编号 | 文本 | 唯一标识 | 150px |
   ```

2. **状态流转用文字描述或代码块**
   ```markdown

   待审核 → 已审核：审批人点击"通过"
   ```

3. **跨模块联动用表格**
   ```markdown
   | 本模块操作 | 联动模块 | 联动效果 |
   |-----------|---------|---------|
   | 销售订单审核通过 | 库存管理 | 可生成出库单 |
   ```

4. **数据模型用表格**
   ```markdown
   | 字段名 | 类型 | 必填 | 说明 |
   |-------|------|------|------|
   | id | bigint | 是 | 主键 |
   ```

## 💻 前端开发规范

### 技术栈
- **HTML5** - 语义化标签
- **CSS3** - 原生 CSS，不使用预处理器
- **原生 JavaScript** - ES6+，不使用框架（Vue/React/Angular）
- **无构建工具** - 直接浏览器打开即可运行

### 代码风格

#### HTML
```html
<!-- 使用 4 空格缩进 -->
<!-- 属性按字母顺序排列 -->
<!-- 类名使用 kebab-case -->
<div class="page-header">
    <h2>销售订单</h2>
    <button class="btn btn-primary">新建</button>
</div>
```

#### CSS
```css
/* 使用 4 空格缩进 */
/* 选择器按层级分组 */
/* 属性按字母顺序排列 */
.btn {
    background: #1890ff;
    border: 1px solid #1890ff;
    border-radius: 4px;
    color: #fff;
    cursor: pointer;
    font-size: 13px;
    padding: 6px 16px;
}
```

#### JavaScript
```javascript
// 使用 4 空格缩进
// 函数使用小驼峰命名
// 常量使用全大写下划线
// 注释使用中文

/**
 * 显示指定模块
 * @param {string} moduleId - 模块ID
 */
function showModule(moduleId) {
    // 隐藏所有模块
    document.querySelectorAll('.module').forEach(function(m) {
        m.classList.add('hidden');
    });

    // 显示目标模块
    var target = document.getElementById(moduleId);
    if (target) {
        target.classList.remove('hidden');
    }
}
```

### UI 设计规范

| 警告色 | `#faad14`（橙色） |
| 错误色 | `#f5222d`（红色） |
| 边框色 | `#d9d9d9` |
| 背景色 | `#f0f2f5` |
| 文字主色 | `#262626` |
| 文字次色 | `#595959` |
| 圆角 | `4px` |
| 阴影 | `0 2px 8px rgba(0,0,0,0.08)` |

### 页面结构

每个模块的预览页面包含：

1. **列表页** - 数据列表 + 查询条件 + 操作按钮
2. **表单页** - 新建/编辑表单
3. **详情页** - 只读详情展示

页面切换通过 `showModule(moduleId)` 函数实现，不使用路由。

## 🔧 开发流程

### 新增模块时的标准流程

1. **阅读现有文档**
   - 先阅读 `ERP_需求文档/01_销售管理.md` 了解文档结构
   - 再阅读 `ERP_前端预览/` 下的代码了解前端结构

2. **编写需求文档**
   - 复制 `01_销售管理.md` 的框架到目标模块
   - 按照业务需求填充内容
   - 保持表格格式一致

3. **开发前端预览**
   - 在 `index.html` 中添加新模块的导航
   - 添加对应的 section 内容
   - 确保样式与现有页面一致

4. **本地验证**
   - 启动本地服务器
   - 检查页面显示是否正常
   - 检查页面切换是否流畅

5. **提交代码**
   - 提交信息使用中文
   - 格式：`模块名：描述`
   - 示例：`采购管理：完成需求文档和前端预览`

## ❌ 禁止事项

1. **不要擅自修改项目结构**
   - 不要新建目录层级
   - 不要修改现有文件名
   - 不要引入新的技术栈

2. **不要过度设计**
   - 不要为简单功能写复杂代码
   - 不要引入设计模式除非必要
   - 不要写未在需求中提到的功能

3. **不要假设需求**
   - 不确定的地方先问用户
   - 不要替用户做决定
   - 不要省略需求中的细节

4. **不要使用英文注释**
   - 所有注释必须使用中文
   - 文档必须使用中文
   - 提交信息必须使用中文
