# Trae AI 开发规则

> 本文档用于 Trae IDE 中的 AI 助手，约束其在处理本项目时的行为。

---

## 📋 强制规则（必须遵守）

### 1. 语言规范
```
优先级：最高
规则：所有输出必须使用中文
适用：代码注释、文档、提交信息、变量命名、用户提示
```

### 2. 需求文档优先原则
```
优先级：最高
规则1：项目核心目标是梳理业务需求，需求文档是主要输出物
规则2：前端预览仅为辅助验证工具，不应优先于需求文档
规则3：任何代码修改前，必须先阅读对应模块的 MD 需求文档
路径：ERP_需求文档/*.md
```

### 3. 需求探索与确认规范
```
优先级：最高
规则1：收到需求相关问题时，必须先探索现有文档和代码
规则2：对于模糊、歧义或冲突的需求，必须与用户多次交互确认
规则3：未确认清楚的需求，不得进入文档更新或前端实现阶段
规则4：每次确认不超过4个问题，避免信息过载
```

### 4. 函数复杂度
```
优先级：高
规则：单个函数不超过 20 行（不含注释和空行）
检查：每次生成代码后自检
```

### 5. 结构一致性
```
优先级：高
规则：新增模块必须遵循现有结构和命名规范
禁止：擅自修改目录结构、引入新技术栈
```

---

## 🎨 前端开发规范

### 技术约束
- HTML：语义化标签，4空格缩进
- CSS：原生 CSS，不使用预处理器，参考 Ant Design 配色
- JS：原生 JavaScript，不使用 Vue/React/Angular
- 工具：无构建工具，浏览器直接打开即可运行

### UI 配色规范
```css
:root {
    --primary: #1890ff;      /* 主色调 */
    --success: #52c41a;      /* 成功 */
    --warning: #faad14;      /* 警告 */
    --error: #f5222d;        /* 错误 */
    --border: #d9d9d9;       /* 边框 */
    --bg: #f0f2f5;           /* 背景 */
    --text-primary: #262626; /* 主文字 */
    --text-secondary: #595959; /* 次文字 */
    --radius: 4px;           /* 圆角 */
}
```

### 页面结构模板
每个模块包含三个视图：
1. 列表页（`#module-list`）- 查询 + 表格 + 分页
2. 表单页（`#module-create`）- 新建/编辑表单
3. 详情页（`#module-detail`）- 只读详情

页面切换函数：
```javascript
function showModule(moduleId) {
    document.querySelectorAll('.module').forEach(m => m.classList.add('hidden'));
    document.getElementById(moduleId)?.classList.remove('hidden');
}
```

## ⭐ 标准需求处理流程（强制执行）

当用户输入任何需求相关内容时，必须严格按照以下流程处理：

**核心原则**：需求文档是主要输出物，前端预览仅为辅助验证工具

### 📋 标准流程（5个步骤）

```
┌─────────────────────────────────────────────────────────────────┐
│  步骤1：探索现有文档                                              │
│  → 读取相关模块的当前需求文档（ERP_需求文档/XX_模块名.md）          │
│  → 理解现有需求结构和已实现内容                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  步骤2：分析对比与差异识别                                         │
│  → 将用户输入的新需求与现有文档进行逐项对比                         │
│  → 识别：新增功能 / 修改现有 / 冲突内容 / 遗漏补充                  │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  步骤3：需求确认（多次交互直到清晰）                                 │
│  → 需求分歧：列出分歧点，提供选项，请用户确认                       │
│  → 需求模糊：要求补充具体示例或说明                                 │
│  → 需求冲突：指出矛盾，说明影响，请用户决定优先级                   │
│  → 分期需求：确认每期范围和优先级                                   │
│  → 多次确认：直到所有疑问都明确才进入下一步                          │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  步骤4：完善需求文档（核心输出）                                     │
│  → 按照文档结构规范，将需求合并到现有文档中                         │
│  → 保持表格格式一致，字段说明清晰                                   │
│  → 在文档顶部用注释标记本次更新的内容                               │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  步骤5：按需调整前端预览（辅助验证）                                 │
│  → 仅在用户明确要求或需要可视化确认时调整前端预览                    │
│  → 使用标注框说明动态效果（弹窗、联动、条件显示等）                 │
│  → 预览目的：确认页面布局和流程，非完整功能实现                     │
└─────────────────────────────────────────────────────────────────┘
```

### 🔍 步骤1：探索现有文档

**必须读取的文件**：
- 对应模块的需求文档（如 `ERP_需求文档/01_销售管理.md`）
- 前端预览代码（如 `ERP_前端预览/index.html`）

**探索目的**：
- 理解当前模块已有的需求结构和已实现内容
- 确保新需求与现有内容兼容
- 保持文档风格和格式一致
- 识别潜在的冲突或重复

### 📊 步骤2：分析对比与差异识别

**对比维度**：

| 对比项 | 分析内容 | 处理方式 |
|--------|---------|---------|
| 功能重叠 | 新需求是否与现有功能重复 | 合并或覆盖，需用户确认 |
| 字段冲突 | 同一字段的定义是否矛盾 | 列出差异，请用户选择 |
| 流程变更 | 是否修改现有业务流程 | 明确变更范围和影响 |
| 新增功能 | 完全独立的新功能 | 直接添加 |
| 遗漏补充 | 现有功能的不完整描述 | 补充完善 |

### 💬 步骤3：需求确认（多次交互）

**必须确认的情况**：

| 场景 | 必须询问的内容 | 询问方式 |
|------|--------------|---------|
| 需求描述不清晰 | 补充业务场景、使用者、触发条件 | 提供具体示例 |
| 需求存在歧义 | 明确用户意图 | 列出所有可能的理解供选择 |
| 与现有文档矛盾 | 确认以哪个为准 | 指出矛盾，说明各自优缺点 |
| 涉及分期实现 | 每期的功能边界 | 用表格列出分期计划 |
| 涉及技术实现 | 实现方案选择 | 说明各方案特点，不替用户决定 |
| 涉及其他模块 | 跨模块影响范围 | 列出涉及的模块和数据流转 |

**确认原则**：
- 一次最多 4 个问题，避免信息过载
- 问题要具体、有选项、有上下文
- 使用表格和列表让问题清晰
- **未确认清楚绝不进入下一步**

### 📝 步骤4：完善需求文档（核心输出）

**合并原则**：
- 新增功能：添加到对应章节末尾
- 修改功能：覆盖原内容，注释说明修改点
- 删除功能：用删除线标注，等待用户确认

**文档更新格式**：
```markdown
## 本次更新说明（新增）

### 2026-05-25 更新
- 新增：XXXX功能（用户输入来源）
- 修改：YYYY字段定义（原定义 → 新定义）
- 补充：ZZZZ业务流程说明

---

## 正文内容（保持原有结构）
...
```

### 🎨 步骤5：按需调整前端预览（辅助验证）

**调整条件**（满足任一条件即可）：
- 用户明确要求查看预览效果
- 新增或修改了页面结构
- 需要可视化确认功能流程
- 用户要求验证需求理解是否正确

**不调整的情况**：
- 纯文档整理，不涉及页面变更
- 用户未要求预览
- 仅补充说明性内容

### 6. 前端预览规范
前端预览页面必须遵循以下原则：

| 原则 | 说明 |
|------|------|
| 简单交互 | 仅实现最基本的页面切换，不做真实的数据交互 |
| 粗粒度原型 | 只展示页面结构和关键元素，不追求像素级细节 |
| 纯静态展示 | 不依赖后端 API，所有数据用硬编码模拟 |
| 快速验证 | 目的是让用户快速确认页面布局和流程是否符合预期 |

**动态效果标注规范**：

当某些需求效果难以通过静态页面直接展示时（如弹窗、联动、异步加载等），必须使用**标注框**的形式在页面上说明预期效果：

| 需要标注的场景 | 标注内容示例 |
|---------------|-------------|
| 弹窗效果 | 【弹窗说明】点击后打开新建订单弹窗，自动填充当前订单的客户信息 |
| 联动效果 | 【联动说明】选择客户后自动带出联系人、电话字段 |
| 异步加载 | 【异步说明】商品列表支持搜索，延迟加载商品数据 |
| 条件显示 | 【条件说明】仅在订单状态为"已审核"时显示"生成出库单"按钮 |

**标注框样式要求**：
```css
.annotation-box {
    background: #fffbe6;      /* 浅黄色背景 */
    border: 1px solid #ffe58f; /* 黄色边框 */
    border-radius: 4px;
    padding: 12px 16px;
    margin: 8px 0;
    font-size: 13px;
    color: #d48806;            /* 深黄色文字 */
}
```

**标注框位置**：
- 紧跟在触发元素下方或旁边
- 保持与周围元素的视觉关联性
- 不遮挡主要内容

## 📝 需求文档规范

### 文档结构
每个模块文档必须包含：
1. 模块概述
2. 页面结构（页签定义 + 每个页签的详细设计）
3. 业务逻辑（数据来源、状态、流转、生命周期、跨模块联动）
4. 表单设计
5. 数据模型

### 表格格式
```markdown
| 字段名 | 类型 | 说明 | 宽度建议 |
|-------|------|------|---------|
| 订单编号 | 文本 | 唯一标识 | 150px |
```

## 🔄 开发流程

### 新增模块步骤
1. 阅读 `ERP_需求文档/01_销售管理.md` 了解结构
2. 阅读 `ERP_前端预览/` 代码了解前端结构
3. 编写需求文档（复制框架并填充）
4. 开发前端预览（添加导航 + section）
5. 本地验证（启动服务器检查）
6. 提交代码（提交信息使用中文）

## ❌ 禁止行为

| 禁止项 | 说明 |
|--------|------|
| 擅自改结构 | 不新建目录、不改文件名、不引入新技术 |
| 过度设计 | 不为简单功能写复杂代码 |
| 假设需求 | 不确定先问用户，不替用户决定 |
| 英文注释 | 所有注释必须使用中文 |
